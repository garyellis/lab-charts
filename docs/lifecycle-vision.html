Platform Services Lifecycle Automation // zero-touch patching

The goal: an upstream chart or image release lands in prod with no human doing anything except reviewing PRs. Every hop is automated, gated by tests, and recorded in an audit trail. This page maps the vision end-to-end and shows which pieces exist in lab-charts + chart-manager today and which are still on paper.

3The promotion ladder

Steps 5–7 above are one hop. The ladder repeats them per environment until prod is rolled out. Each hop is gated: Ready + helm test pass + soak window elapsed + version-forward. A failure stops the line at that env — nothing past it moves.

dev

first landing zone
tracks lab-charts closely
⟶
rollout ✓ · helm test ✓ · soak ✓
→ PR to uat (human merges)

uat

longer soak
pinned versions
⟶
rollout ✓ · helm test ✓ · soak ✓
→ PR to prod (human merges)

prod

reached_prod event closes the loop
duration is measured end-to-end
Zero-touch means zero typing, not zero eyes. The only human actions in the whole loop are PR reviews — the bump PR in lab-charts and one promotion PR per env hop. Detection, testing, PR shaping, validation, and bookkeeping are all machine work. Auto-merge is deliberately off the table until a CI-equivalent gate runs against the GitOps PRs themselves.

4The quality gates

Three layers of defense between "new version exists" and "running in prod". The first two run in lab-charts CI today on every PR; the third runs per env after rollout.

Static analysis Built

  • helm template render per chart × env (values composed per validate-spec.yaml)
  • kubeconform — Kubernetes schema validation
  • kyverno — policy-as-code (e.g. forbid LoadBalancer)
  • runs in seconds; full fan-out across all charts/envs

Sandbox integration test Built

  • ephemeral kind cluster per changed chart (CI matrix)
  • real dependency install plan (cilium CNI, istio, …)
  • real helm install + helm test hooks + checks
  • same command locally: mise run kind-test
  • planned: upgrade-mode (install baseline → upgrade to PR)

In-env validation Partial

  • HelmRelease Ready + observedGeneration checks (helmrelease monitor)
  • helm test in the live env (helmrelease test)
  • planned: soak window + digest-level dedupe (watcher contract)
  • planned: k8s Events + Prometheus promotion metrics

5The audit trail spine

Every stage emits a typed lifecycle event keyed by correlation id, so one upstream release is traceable from bump PR to prod. Capture is built — chart-manager events build|promote with DynamoDB and Cosmos DB backends. The query layer (dashboards, lead-time metrics) is the deferred Phase 5.

detected→ flux_pr_open→ awaiting_merge→ waiting_for_rollout→ rollout_complete→ helm_test_running→ helm_test_passed→ promoted→ reached_prod
helm_test_failed abandoned ← failure phases; captured so failed promotions are first-class data, not lost logs
Answers the DESIGN.md questions directly: how long from renovate PR to all envs? and which promotions failed, where, and why? Per-env timestamps on one correlation id make lead time a query, not an archaeology project.

7Program phases

The phase framing from the planning docs, annotated with where we actually are.

1

Phase 1 — Supply chain Planned

Vendor image → trusted artifact. Ingestion watcher, cosign/SLSA verification, approved-channel tags, auto bump PRs. Design complete (psl-image-bump-plan.md); vendor annotation contract needs sign-off.

2

Phase 2 — Chart packaging Built

~25 wrapper charts with per-chart validate/test specs. This is the substrate everything else operates on.

3

Phase 3 — Validate + sandbox CI Built

Static analysis and kind integration tests, identical locally and in CI. Remaining: upgrade-mode matrix (psl-ci-test) and the publish workflow.

4

Phase 4 — Delivery + promotion Partial

The per-hop mechanics are built as CLI commands (helmrelease promote / monitor / test). The testbed, promotion contract, and always-on controller that chain the hops together are designed, not built. This is the current frontier.

5

Phase 5 — Lifecycle audit Partial

Event capture is ahead of schedule — the writer, phase model, and DynamoDB/Cosmos backends exist and the promote path already emits. The query/reporting layer (deploy counts, lead time, failure rates) is deferred by design.
